## 2. 产品背景

犬只管理系统已经存在多个协同端口，包括犬主人端、城管端、服务终端、运营后台、业务服务 API 和数据清洗 / 算法层。

监管后台端的职责不是重复其他端口能力，而是汇总犬只治理过程中的关键数据，形成面向监管人员的全局治理工作台。

监管后台端一期重点解决以下问题：

- 看不清全局治理态势
- 找不到重点风险和关注对象
- 追不到任务闭环和处理结果
- 难以把分散数据组织成可汇报的治理成效

---

## 3. 产品定位

监管后台端一期定位为：**犬只治理生态中的 PC 端治理中枢**。

一期不做全功能后台，而是做一个四页工作台，支撑监管人员完成以下四类核心动作：

```text
看全局
看地图
查对象
管积分
```

对应四个一级页面：

```text
治理驾驶舱
地图工作台
对象档案
积分管理
```

---

## 4. 一期目标

### 4.1 产品目标

1. 建立监管端统一入口，承接全局治理视角
2. 让监管人员快速识别风险、空间分布和重点对象
3. 在不新增复杂后台结构的前提下，形成可演示、可评审、可实现的一期方案
4. 把正向激励纳入监管端主线，不让系统只剩“异常、处罚、报警”

### 4.2 一期价值

一期不以“功能最多”为目标，而以“入口少、聚合强、能落地、能演示”为目标。

## 7. 关键业务原则

### 7.1 监管端是治理中枢，不替代其他端口

监管端负责汇聚关键数据、展示治理态势、追踪风险和闭环，不替代：

- 犬主人端的用户服务能力
- 城管端的一线核验与任务执行
- 服务终端的积分兑换与核销
- 运营后台的经营分析与商家运营

### 7.2 犬只是监管主对象

监管端的核心主体是犬只，不单独建设犬主一级页面。

### 7.3 绑定关系原则

- 狗牌与犬只绑定关系由政府侧统一维护
- 犬主人端负责主手机号与已发放狗狗 / 狗牌的绑定或认领
- 监管端只展示绑定结果、积分归属和异常兜底信息

### 7.4 积分归属原则

- 积分归属于犬只ID或狗牌编号
- 主手机号是犬主账户和权益账户
- 其他联系方式只是附属联系方式

### 7.5 事件承接原则

一期不建设独立事件中心。

事件能力通过以下三处共同承接：

- 治理驾驶舱的风险提醒
- 地图工作台的事件点位与轻量闭环查看
- 对象档案中的异常记录

其中地图工作台是一期开启事件空间定位和轻量闭环查看的主要页面。

## 9. 页面需求

## 9.1 治理驾驶舱

### 页面目标

- 总览整体治理态势
- 快速发现风险与重点关注对象
- 观察近期趋势和治理成效
- 向地图、档案、积分页面分发流量

### 页面结构

```text
页面标题区
核心指标区
空间态势区
风险提醒区
趋势分析区
治理成效区
```

### 功能要求

#### 9.1.1 页面标题区

- 支持页面标题展示
- 支持时间范围切换：今日 / 近7日 / 近30日
- 支持区域筛选：全市 / 区县 / 街道
- 展示数据更新时间

#### 9.1.2 核心指标区

建议展示以下指标：

```text
在册犬只数
设备在线率
今日新增异常
待处理事件
积分参与率
服务核销次数
```

每个指标卡需支持：

- 名称
- 数值
- 趋势变化
- 点击跳转

跳转要求：

- 今日新增异常 -> 地图工作台
- 待处理事件 -> 地图工作台
- 在册犬只数 -> 对象档案
- 积分参与率 -> 积分管理

#### 9.1.3 风险提醒区

每条风险提醒至少包含：

- 风险类型
- 风险等级
- 关联区域
- 关联对象
- 发生时间
- 快捷入口：查看地图 / 查看档案

一期风险表达按规则预警或轻量分析处理，不包装为复杂 AI 推理系统。

#### 9.1.4 空间态势区

用于快速呈现空间维度的待关注信息。

建议承载：

- 高风险事件
- 长时间未闭环事项
- 设备离线集中区域
- 积分低参与区域

#### 9.1.5 趋势分析区

至少支持 2 到 3 组轻量趋势展示：

- 异常事件趋势
- 闭环处理趋势
- 积分参与趋势

#### 9.1.6 治理成效区

至少支持展示：

- 异常下降趋势
- 设备在线改善情况
- 积分参与提升情况
- 服务核销活跃情况

## 9.2 地图工作台

### 页面目标

- 看风险在哪里
- 看犬只在哪里
- 看事件在哪里
- 看城管辖区覆盖在哪里
- 看服务商覆盖在哪里
- 承接一期轻量事件闭环查看

### 页面结构

```text
页面标题区
筛选工具栏
左侧图层面板
中间地图区域
右侧详情抽屉
底部统计条（可选）
```

### 功能要求

#### 9.2.1 页面标题区

- 展示页面标题
- 展示当前区域
- 支持时间筛选
- 支持一键重置筛选

#### 9.2.2 筛选工具栏

支持以下筛选项：

- 对象类型：犬只 / 事件 / 城管辖区 / 服务商
- 风险等级：高 / 中 / 低
- 状态：正常 / 异常 / 待关注
- 时间范围
- 区域范围

#### 9.2.3 图层面板

一期默认图层：

```text
犬只点位
事件点位
城管任务区域 / 负责辖区
服务商点位
重点区域 / 禁遛区域
```

一期明确限制：

- 不展示城管实时定位
- 不提供轨迹回放

#### 9.2.4 地图主体

地图需区分不同对象类型：

- 犬只：常规圆点
- 事件：高亮风险点
- 城管辖区：轮廓区块
- 服务商：固定服务点图标
- 重点区域：半透明面

#### 9.2.5 详情抽屉

点击点位后，需根据对象类型展示差异化信息。

犬只点位信息：

- 犬只名称 / ID
- 状态
- 主手机号摘要
- 最近定位时间
- 查看档案入口

事件点位信息：

- 事件名称
- 风险等级
- 发生时间
- 关联犬只
- 处理状态
- 处理人 / 负责辖区
- 查看犬只档案入口

服务商点位信息：

- 名称
- 类型
- 合作状态
- 最近核销时间
- 查看服务商档案入口

#### 9.2.6 底部统计条

可选展示：

- 犬只数
- 事件数
- 高风险数
- 服务点数

### 轻量事件闭环要求

一期不建设地图内完整事件处理流转，只要求支持：

- 查看事件摘要
- 查看事件当前状态
- 查看处理人或负责辖区
- 跳转关联犬只档案继续判断

## 9.3 对象档案

### 页面目标

- 承接驾驶舱、地图和全局搜索进入的对象查询
- 支撑监管人员查看主体详情
- 与积分管理形成关联跳转

### 页面结构

```text
页面标题区
搜索与筛选区
对象类型 tab
左侧列表区
右侧详情区
```

对象类型 tab 包含：

```text
犬只
城管
服务商 / 合作商
```

默认进入“犬只”。

### 功能要求

#### 9.3.1 搜索与筛选

支持按以下信息检索：

- 犬只 ID
- 主手机号
- 城管编号
- 服务商名称

#### 9.3.2 犬只列表

列表项至少展示：

- 名称
- 犬只 ID
- 状态标签
- 所属区域
- 最近更新时间

#### 9.3.3 犬只详情

需展示以下模块：

```text
基础信息
狗牌 / 设备信息
主手机号与绑定状态
积分摘要
定位状态
核验记录
行为记录
异常记录
```

补充说明：

- 狗牌绑定来源：政府统一绑定
- 主号绑定来源：犬主人端绑定 / 认领

#### 9.3.4 城管列表与详情

城管列表项展示：

- 姓名
- 编号
- 所属辖区
- 近期任务量摘要

城管详情展示：

- 基础信息
- 所属辖区
- 任务负责范围
- 近期核验记录
- 近期任务处理情况
- 在管区域摘要

#### 9.3.5 服务商列表与详情

服务商列表项展示：

- 名称
- 类型
- 合作状态
- 最近核销时间

服务商详情展示：

- 基础信息
- 位置
- 合作状态
- 积分核销次数
- 最近核销时间
- 服务覆盖摘要

一期不做经营分析大盘。

### 页面目标

- 看规则是否合理
- 看账户是否健康
- 看流水是否清晰
- 看激励是否产生治理价值

### 页面结构

```text
页面标题区
指标摘要区
积分规则区
积分账户区
积分流水区
积分效果区
```

### 功能要求

#### 9.4.1 指标摘要区

建议展示：

```text
积分参与率
活跃账户数
本期发放积分
本期核销积分
服务核销次数
异常账户数
```

#### 9.4.2 积分规则区

每条规则至少展示：

- 规则名称
- 触发行为
- 发放积分
- 状态
- 适用范围

一期只展示，不做复杂规则编辑器。

#### 9.4.3 积分账户区

账户以主手机号为核心，至少展示：

- 主手机号
- 关联犬只数
- 当前积分余额
- 最近活跃时间
- 状态

#### 9.4.4 积分流水区

至少展示：

- 时间
- 主手机号
- 关联行为
- 积分变化
- 来源类型

#### 9.4.5 积分效果区

至少支持观察：

- 不开放手动调整积分

4. 服务商能与积分核销关系挂钩

## 12. 非功能与体验要求

### 12.1 体验要求

- 风格应专业、克制、偏治理感
- 风险色明确但不能喧宾夺主
- 页面信息密度高但结构清楚
- 支持中高保真 HTML 原型表达
- 统一采用 CDHT 橙色作为品牌主色，转译为现代、温和、可信的治理后台视觉系统
- 以 Airtable 式结构化数据后台作为主风格参考，Coinbase、HashiCorp 等外部风格仅作为页面级信息组织参考，不引入独立色调或组件体系
- 四个一级页面需共享统一 Token、按钮、标签、卡片、表格、筛选器和抽屉气质，避免形成多套视觉主题
- 避免大面积纯黑标题、过硬边框、0 圆角和冷淡背景，评审演示场景需体现现代感与品牌温度

### 12.2 一期实现要求

- 优先保证四页结构完整
- 优先保证页面跳转和演示闭环
- 优先保证地图、档案、积分之间的逻辑一致

## 14. 验收标准

一期 PRD 对应的原型 / 页面设计，至少需要满足以下验收点：

1. 一级导航只有四页
2. 驾驶舱能体现全局态势、风险和成效
3. 地图工作台能看点位、筛选对象并查看轻量事件闭环信息
4. 对象档案能覆盖犬只、城管、服务商三类主体
5. 积分管理能体现规则、账户、流水和效果
6. 不出现独立事件中心入口
7. 不出现手动积分调整入口
8. 犬只详情中能明确展示狗牌绑定来源和主号绑定来源

```text
监管后台端一期不做全能后台，只做四页工作台：
驾驶舱给方向，
地图给空间，
档案给主体，
积分给闭环。
```
