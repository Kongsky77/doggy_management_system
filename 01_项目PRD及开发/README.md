## 监管后台端当前状态

监管后台端原“四页工作台”方案已于 2026-07-10 废止，现按新产品重新启动定义流程。旧 PRD、设计规范和 HTML 原型仅供历史追溯，不再作为当前执行依据。

当前已确认的新方向为：以大地图作为单页主体，核心用户为领导，监管端只提供查看、筛选、AI 辅助决策以及图片或报告导出。地图只接入电子狗牌犬只定位；默认“实时时间”要求全部可定位犬只以独立气泡显示，选择全部或预设时间段后则隐藏单犬气泡，展示最近 7 天相同时段遛狗热力图。不展示城管信息，不生成街道或社区数量聚合，不接入普通狗牌手机定位。保留服务商、宠物友好乐园和犬只禁入区域图层。区域筛选与时间视图使用独立控件；首屏不单独展示 KPI，区域指标进入右侧详情；右下角只保留放大、缩小和图层入口。顶部提供跨友好乐园与服务商的全局活动列表；AI 深度分析以重点结论和三类时间图表呈现。

- 当前有效 PRD、精简方案和功能说明

- 与当前有效 PRD 对齐的原型说明
- 可直接评审的 HTML 原型

## 管理要求

- PRD 发生变化时，原型应同步调整。
- 如果某份设计稿或讨论稿尚未成为正式需求依据，应移出本目录，放入 `00_辅助资料/`。
- 同一端口的文件前缀编号表示端口编号，不表示文档顺序；例如监管后台端作为第二个端口，其正式 PRD 和原型文件都应使用 `02_` 作为前缀。
- 同一份文档后续更新时，应直接修改文件尾部版本号，例如从 `v1.0` 更新为 `v1.1`，而不是改动前缀编号。
- 文件名应包含业务对象、文档类型和版本号，例如 `02_监管后台端一期PRD_v1.0.md`。
- 当前有效版本应在文档顶部标明版本、日期和状态。
- 已废弃方案优先通过 Git 历史回溯，不在本目录长期堆放多个副本。
