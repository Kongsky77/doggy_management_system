## 1. 设计目标

运营端是超级管理员每天进行跨端数据维护和经营判断的主工作台。界面应让用户快速识别当前工作重点、稳定地处理对象与积分数据，并能随时解释“谁在何时为何修改了什么”。

### 1.1 设计方向

- 气质：专业、温和、结构化、可追溯；
- 受众：高频使用数据表、筛选、表单和记录的超级管理员；
- 记忆点：用一条“跨端运营脉搏”串起驾驶舱的端口动向与待关注事项；
- 约束：宽桌面优先、表格可扫读、编辑操作受控、长字段不溢出、操作历史随时可达。

不使用营销页式大标题、装饰渐变、卡片套卡片或大面积高饱和品牌色。

### 1.2 旧原型复用规则

旧监管端一期原型是本端的通用后台组件参考。运营端直接复用其左侧导航、驾驶舱分层、对象档案 Tab / 表格 / 详情、积分规则与流水表、状态组件及页面切换模式；但必须替换为当前运营端的数据权属、字段、操作和文案。

地图工作台、监管专属事件与任务、城管辖区、领导只读口径及地图导出能力不进入运营端。运营端独立新建 `05_` 原型文件与资产目录，绝不修改旧 `02_` 原型。

## 2. 页面结构

```text
┌──────────────┬───────────────────────────────────────────────────────┐
│ 品牌与产品名 │ 顶栏：全局状态 / 超级管理员 / 当前数据时间             │
│              ├───────────────────────────────────────────────────────┤
│ 运营驾驶舱   │ 页面标题、说明、主操作                                 │
│ 对象档案     │ 筛选与状态切换                                         │
│ 积分管理     │ 摘要指标（按页面需要出现）                             │
│ 活动管理     │ 主图表或主列表                                         │
│              │ 右侧详情抽屉 / 确认弹窗（按需覆盖）                     │
└──────────────┴───────────────────────────────────────────────────────┘
```

### 2.1 宽桌面规则

| 项目 | 1920 主画布 | 1440 兼容画布 |
| --- | ---: | ---: |
| 左侧导航 | 232px | 208px |
| 顶栏高度 | 64px | 64px |
| 内容安全边距 | 24px | 20px |
| 表格最小宽度 | 100% 内容区 | 允许表格内部横向滚动 |
| 详情抽屉宽度 | 480px | 420px |

低于 `1440px` 不继续压缩为移动布局；显示宽桌面访问提示或保留最小画布横向滚动。

### 2.2 左侧导航

- 固定显示品牌标识、产品名“犬只运营管理后台”和四个一级入口；
- 当前页面使用品牌浅色背景、橙色图标和左侧 3px 强调条；
- 导航不承载二级对象 Tab，二级切换位于对应页面内容区；
- 不在一期加入角色管理、通知中心、任务中心或设置入口。

### 2.3 顶栏

顶部仅保留当前数据更新时间、系统状态与“超级管理员”身份信息。无全局搜索、消息中心或复杂面包屑。页面内筛选只影响当前模块，不放入顶栏。

## 4. 公共组件与状态

### 4.1 页面头部

每页使用“标题 + 一句用途说明 + 右侧主操作”的单行结构。主操作最多一个，其他操作放入次级按钮或更多菜单。

| 页面 | 主操作 |
| --- | --- |
| 运营驾驶舱 | 无；只读下钻 |
| 对象档案 | 新增对象 |
| 积分管理 | 人工调整积分 |
| 活动管理 | 新建活动 |

### 4.2 筛选栏

筛选置于页面头部下方，与主列表或图表同一工作区。包含关键词搜索、对象 / 状态 / 时间等当前页面所需条件和“重置”按钮；默认不展开高级筛选。

### 4.3 表格与状态标签

- 表头固定在滚动容器顶部；行高 52–56px；
- 关键对象名左对齐，数值右对齐，时间与状态保持固定列宽；
- 长地址、说明等字段截断为一行，悬停显示完整内容；
- 状态以带文字的轻量胶囊标签表达，不使用大色块；
- 空状态显示原因、一个恢复动作与简洁图标，不用虚构数据补位；
- 加载状态使用骨架，不闪烁整页；失败状态提供重试。

### 4.4 详情抽屉与编辑

点击列表行打开右侧详情抽屉，覆盖内容区而不改变左侧导航与表格宽度。详情顶部显示对象名、状态和关闭按钮；底部固定“编辑”主按钮及必要的次级操作。

编辑在同一抽屉的编辑态或居中表单弹窗完成。危险操作（删除、下线、恢复、扣减积分）必须二次确认，且要求填写原因。保存成功后显示轻提示，并刷新列表、详情和操作历史。

### 4.5 操作历史入口

对象档案和积分管理右下角固定显示悬浮圆形按钮“操作历史”。点击后打开右侧历史面板：

- 默认按当前页面对象类型筛选；若已打开详情，则自动按当前对象筛选；
- 每条记录显示操作类型、对象、操作者、时间、原因与前后值摘要；
- 支持按操作类型、时间和对象状态筛选；
- 历史面板只读，无删除、编辑、导出或补录入口；
- 关闭后保留主页面原有筛选与滚动位置。

活动与区域的操作历史从详情抽屉内的“查看操作记录”进入，避免同屏出现多个悬浮按钮。

## 5. 页面设计

### 5.1 运营驾驶舱

#### 目标与布局

首屏先呈现本周期的跨端运营脉搏，下面按“对象与设备、积分、服务商与活动、待关注事项”排列。驾驶舱是判断与跳转入口，不放置可编辑表单。

```text
页面标题 / 当前周期
跨端运营脉搏：犬主人端｜城管端｜服务端｜监管端
对象与设备趋势      积分经营概览
服务商与活动表现    待关注事项
```

- 跨端运营脉搏使用四个紧凑状态单元，显示活跃量、较上一周期变化和状态；
- 对象与设备、积分、服务商与活动各使用一个主图表或摘要组合，避免每项指标独立成卡；
- 待关注事项按风险优先级列出，点击跳转到对应页面并带入筛选；
- 首屏可展示 4–6 个关键数字，不出现大面积 KPI 墙。

### 5.2 对象档案

#### 页面结构

```text
标题 / 新增对象
对象 Tab：犬只与犬主人｜城管队员｜服务商｜地图区域
关键词、状态与当前对象筛选
对象表格
右侧详情 / 编辑抽屉
右下角操作历史入口
```

#### 各对象列表

| Tab | 默认关键列 | 详情重点 |
| --- | --- | --- |
| 犬只与犬主人 | 犬只姓名、品种、狗牌编号 / 种类、犬主人手机号、当前积分、状态 | 完整犬只字段、设备编号、积分只读摘要、关联流水跳转 |
| 城管队员 | 姓名、队员编号、所属大队、辖区、核销犬只次数、状态 | 基础档案与只读核销累计 |
| 服务商 | 名称、类别、服务商品质、积分核销数、关联活动数、状态 | 地址、品质编辑、只读核销与活动摘要 |
| 地图区域 | 区域名称、类型、所属街道 / 社区、关联活动数、状态 | 边界描述、上线状态、友好区域关联活动 |

新增对象先选择对象类型，再显示对应表单。狗牌种类选择电子狗牌时显示并必填“内部设备编号”；选择普通狗牌时隐藏该字段。积分摘要永远只读，按钮文案为“查看积分流水”，跳转积分管理。

### 5.3 积分管理

#### 页面结构

```text
标题 / 人工调整积分
二级 Tab：积分规则｜积分流水｜经营分析
页面级筛选
当前 Tab 的规则表、流水表或经营图表
右下角操作历史入口
```

| Tab | 主内容 | 关键交互 |
| --- | --- | --- |
| 积分规则 | 规则列表，展示名称、类型、适用对象、积分值、生效时间、状态 | 新增、编辑、启用、停用；已生效历史不回写 |
| 积分流水 | 流水表，展示编号、时间、对象、变动类型、变动值、余额、来源 | 只读查询、跳转对象详情 |
| 经营分析 | 发放、核销、余额与使用率的趋势和结构图 | 时间、对象类型、服务商筛选；分母为零显示明确空状态 |

人工调整使用居中确认表单：先选择对象，再选择发放或扣减、输入积分数、填写必填原因，最后在确认页展示调整前 / 后余额。提交后生成新流水，不提供编辑历史流水的入口。

### 5.4 活动管理

#### 页面结构

```text
标题 / 新建活动
关键词、状态、时间、举办地点筛选
活动列表（名称、状态、时间、地点、关联对象、更新时间）
右侧活动详情 / 编辑抽屉
```

活动状态使用草稿、已上线、已下线、已结束四种文字标签。详情按“基础信息、关联场地与服务商、活动说明、参与提示、状态记录”分组；上线、下线与结束操作固定在详情底部，并要求确认及原因。草稿与下线活动使用弱化状态但仍可通过筛选查询。

## 6. 关键交互与反馈

| 场景 | 交互规则 |
| --- | --- |
| 页面切换 | 导航立即切换高亮，内容区 160–200ms 淡入；不使用装饰性大动画 |
| 列表查询 | 输入停止 300ms 后更新，加载时保留表头与骨架行 |
| 详情打开 | 右侧抽屉 180–220ms 滑入，焦点进入抽屉标题 |
| 保存成功 | 显示简洁成功提示，列表和详情同步更新 |
| 删除 / 下线 | 确认弹窗中明确说明“保留历史数据”，原因必填 |
| 恢复 / 重新上线 | 同样需要原因与确认，成功后恢复默认有效列表可见性 |
| 失败 | 说明失败原因，保留用户已输入内容并提供重试 |

所有按钮、Tab、筛选、表格行、抽屉关闭和悬浮历史入口必须支持键盘焦点。`Esc` 优先关闭最上层抽屉或弹窗；焦点回到触发元素。

## 7. 原型数据与验收

### 7.1 演示数据约束

- 演示犬只、狗牌、设备编号、犬主人手机号与积分流水应相互对应；
- 电子狗牌与普通狗牌分别覆盖，验证条件字段显示；
- 至少包含有效、已删除、已上线、已下线、草稿、已结束等状态；
- 人工积分调整必须能生成一条带原因的新流水和操作历史；
- 服务商品质、活动与友好区域关联应可被交叉查看。

### 7.2 设计验收

1. 四个页面共享左侧导航、顶栏、颜色、状态标签、表格、抽屉与确认交互。
2. `1920 × 1080` 与最小 `1440px` 宽度下，导航、表格、抽屉和悬浮入口无重叠或文字溢出。
3. 主色仅用于品牌与关键操作；成功、警示、风险均有独立语义与文字标签。
4. 对象档案与积分管理的操作历史入口固定在右下角，历史记录不可编辑或删除。
5. 所有删除、下线、恢复、重新上线和积分扣减均要求确认与原因。
6. 对象档案不出现可编辑积分控件；积分流水不出现编辑或删除入口。
7. 监管端只读的数据边界通过服务商品质、活动和区域的只读说明体现，不在运营端增加监管端操作入口。

## 8. 一句话设计定义

> 运营后台端是一个暖白、紧凑且可追溯的运营工作台：左侧稳定切换模块，中央高效处理数据，右侧按需查看与编辑，右下角随时追溯操作历史。
