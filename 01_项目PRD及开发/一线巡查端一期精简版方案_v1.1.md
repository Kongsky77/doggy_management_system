# 一线巡查端一期精简版方案

## 一、一期产品定位

一线巡查端是面向城管队员的移动巡查工具，一期核心目标是：

> **帮助一线人员快速核验犬只身份、查看基础状态与最近位置，并完成简单任务处理。**

一期不追求完整监管能力，不做复杂地图巡查、不做权限管理系统、不展示设备原始数据、不接入疫苗体检记录。

一期关键词：

```text
查得到
看得懂
能处理
少操作
```

---

# 二、一期核心功能范围

## 一期保留功能

| 模块   | 功能说明                   |
| ---- | ---------------------- |
| 登录   | 微信授权登录，校验是否绑定城管人员身份    |
| 首页   | 展示人员信息、快捷入口、今日简报、待处理任务 |
| 扫码核验 | 扫描狗牌二维码，识别犬只信息         |
| 手动查询 | 输入犬只 ID / 狗牌编号查询犬只     |
| 犬只详情 | 查看犬只、犬主、狗牌、登记状态、定位摘要   |
| 位置查看 | 查看单只犬最近位置              |
| 任务消息 | 查看任务 / 异常提醒，进行简单处理     |
| 我的   | 查看个人信息，退出登录            |

原全量方案中已经包含登录、首页、扫码、手动查询、犬只详情、地图、消息、我的等页面结构，本次一期版本在此基础上进行精简。

---

# 三、一期不做功能

以下功能一期暂不做，放到二期或后续：

| 功能            | 不做原因                               |
| ------------- | ---------------------------------- |
| 权限管理系统        | 一线端不做复杂权限配置，只做身份绑定校验               |
| 复杂地图巡查        | 一期只做单犬位置查看，不做全辖区地图                 |
| 重点区域 / 禁遛区域边界 | 需要地图规则配置，先后置                       |
| 复杂筛选          | 一期场景简单，避免增加操作成本                    |
| 轨迹回放          | 数据展示和地图开发成本较高                      |
| AI 异常识别       | 依赖数据积累和算法能力                        |
| 疫苗体检记录        | 当前拿不到可靠数据，且非一期核心                   |
| 设备原始数据展示      | 城管端不需要理解 rssi、accuracy、heading 等字段 |
| 拍照上传          | 可作为二期增强功能                          |
| 完整巡查记录列表      | 一期只做简单处理记录或后台自动留痕                  |

一句话：

> **一期只做“查犬 + 看位置 + 处理任务”，其他都先砍。**

# 四、一期页面结构

一期建议保留 8 个页面：

```text
1. 登录页
2. 首页
3. 扫码页
4. 手动查询页
5. 犬只详情页
6. 单犬位置页
7. 任务消息页
8. 我的页
```

底部导航建议：

```text
首页｜扫码｜任务｜我的
```

暂时不把“地图”作为一级 Tab，避免一期地图能力不足却被用户当成核心功能。人类很擅长把半成品放首页，然后祈祷用户别点，别这样。

# 五、页面精简设计

## 1. 登录页

### 保留内容

```text
系统名称
一句话说明
微信授权登录按钮
授权失败提示
未绑定身份提示
```

### 登录逻辑

```text
微信授权
  ↓
校验是否绑定城管人员身份
  ↓
绑定成功：进入首页
未绑定：提示联系管理员
```

### 不做

```text
角色权限配置
菜单权限
字段权限
复杂组织架构
```

---

## 2. 首页

首页目标：让队员知道自己是谁、今天要做什么、从哪里开始操作。

### 页面内容

```text
张三
锦江区城管队｜XX社区

快捷操作
[扫一扫] [手动输入]

今日简报
今日核验：12
待处理任务：5
已处理：8

待处理任务
1. 犬只进入重点区域提醒
2. 设备离线提醒
3. 投诉事件待处理

[查看更多]
```

### 首页保留

- 队员姓名

- 所属部门 / 所属辖区

- 扫码入口

- 手动查询入口

- 今日核验数

- 待处理任务数

- 已处理任务数

- 最新 3 条任务

### 首页砍掉

- 新录入犬只数

- 复杂数据看板

- 全量异常统计

- 地图定位快捷大模块

- 多种复杂消息分类

---

## 3. 扫码页

扫码是一期核心功能，必须保留。

### 功能

```text
调用微信扫码
识别狗牌二维码
扫码成功后展示结果
扫码失败提示
支持重新扫码
支持跳转手动输入
```

### 扫码成功结果

```text
扫码成功

犬只：豆豆
犬只 ID：D023
狗牌编号：P10293
狗牌类型：电子狗牌 / 普通狗牌
登记状态：已登记

[查看详情] [继续扫码]
```

### 异常状态

```text
二维码无效
犬只不存在
狗牌未绑定
网络错误
无权限 / 未绑定身份
```

---

## 4. 手动查询页

手动查询作为扫码失败兜底。

### 页面内容

```text
请输入犬只 ID / 狗牌编号

[输入框]
[查询]
```

### 查询结果

```text
查询成功：进入犬只详情
未找到：提示未查询到犬只信息
格式错误：提示输入格式错误
```

### 一期只支持

```text
犬只 ID
狗牌编号
```

不支持犬主手机号、模糊搜索、地点搜索。先别把搜索框做成许愿池。

---

## 5. 犬只详情页

详情页一期只展示现场判断需要的信息。

### 页面目标

让城管队员快速知道：

```text

是否登记？
是谁的狗？
有没有最近位置？
我能做什么？
```

# 七、一期数据展示原则

## 1. 不展示设备原始数据

电子狗牌上传的数据包括经纬度、定位时间、精度、速度、方向、电量、信号强度、定位类型、离线缓存、断网补传等。

一期前端只展示处理后的业务信息：

```text
最近位置
最近定位时间
定位来源
定位状态
```

设备原始数据用于后台计算和地图定位，不在一线端详情页展开。

---

## 2. 不展示拿不到的数据

疫苗体检记录一期不做，因为当前暂无可靠数据来源。  
拿不到数据还硬做页面，就是给自己挖坑再铺瓷砖，精致但愚蠢。

---

## 3. 不做复杂权限管理

一期只做身份绑定校验：

```text
能不能登录
是谁
属于哪个单位 / 辖区
```

不做角色权限配置、字段权限、菜单权限。

# 八、一期 / 二期划分

## 一期：基础核验闭环

目标：

> 让城管队员能快速查犬、看基础信息、查看最近位置、处理简单任务。

范围：

```text
登录
首页
扫码
手动查询
犬只详情
单犬位置
任务消息
我的
```

---

## 二期：地图巡查能力

目标：

> 支持附近犬只、异常事件、重点区域的地图化巡查。

范围：

```text
完整巡查地图
附近犬只点位
遛狗中犬只
重点区域 / 禁遛区域
地图筛选
导航前往
异常上报
```

---

## 三期：智能监管能力

目标：

> 基于数据分析和算法形成风险识别能力。

范围：

```text
轨迹回放
风险等级
AI 异常识别
设备健康分析
巡查绩效统计
智能任务推荐
```

---

# 九、可以直接对领导说的版本

> 一期建议聚焦“一线核验闭环”，只做登录、扫码、手动查询、犬只详情、单犬位置查看、任务消息和我的页面。
> 
> 不做复杂权限管理，不展示设备原始数据，不做疫苗体检记录，也不做完整巡查地图。电子狗牌数据只用于地图定位和后台判断，一线端只展示最近位置、定位来源和定位状态。
> 
> 一期目标是让城管队员能快速查犬、看懂基础信息、查看最近位置，并完成简单任务处理。二期再扩展完整地图巡查、重点区域、轨迹回放和异常上报。

---

# 十、最终一句话

> **一期只做“查犬、看位置、处理任务”；二期再做“巡查地图”；三期再做“智能分析”。**
