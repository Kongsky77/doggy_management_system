## 2. 产品背景

犬只管理系统包括犬主人端、城管端、服务端、监管端等多个业务端口。监管端已收敛为只读地图决策界面，不再承担对象、积分、服务商及活动的管理职责。

运营后台端需要成为上述运营数据的唯一管理入口：超级管理员在此了解各端数据动向、维护对象与区域、管理活动、制定积分规则、调整积分，并通过完整留痕支撑运营解释与追溯。

## 3. 产品定位

> 面向超级管理员的跨端运营管理后台：用驾驶舱定位工作重点，用对象档案统一维护业务数据，用积分管理解释和优化积分体系，用活动管理维护运营活动。

运营端维护后的有效数据由其他端按职责读取：监管端可只读消费服务商品质、区域及活动等相关数据；其他端不得提供相同数据的管理操作。

## 4. 用户、权限与数据边界

### 4.1 一期用户

一期只有一种使用者：超级管理员。默认其已完成登录并拥有本 PRD 内的全部操作权限。

一期不建设运营专员、只读人员等角色配置、授权界面、审批流或细粒度数据权限；后续增加多角色时，不改变本期确定的数据归属和页面职责。

### 4.2 数据权属

运营端是犬只、关联犬主人信息、城管队员、服务商、地图区域、活动及积分运营数据的唯一管理端。

其他端可产生业务使用数据或读取运营端发布的有效数据，但不得在自身端口新增、编辑、删除或下线上述运营数据。历史核销、活动参与和积分流水等业务事实不得被改写。

### 4.3 删除与下线

面向用户的“删除”统一为逻辑删除或下线：

- 犬只、城管队员、服务商等对象删除后不显示在默认有效列表；
- 禁入区域、友好区域和活动以“下线”表示停止生效；
- 已删除或下线数据可通过状态筛选查询；
- 历史积分流水、核销记录、活动记录和操作日志永久保留；
- 恢复对象或重新上线时必须记录操作原因。

## 5. 一期目标与非目标

### 5.1 目标

1. 让超级管理员快速判断跨端运营重点。
2. 为四类对象及地图区域提供统一、可追溯的维护入口。
3. 让积分规则、人工调整、历史流水和经营分析有唯一入口与一致口径。
4. 在运营端创建和维护活动，并向监管端提供只读活动数据。
5. 为高风险管理操作建立可查询的操作历史。

### 5.2 不纳入一期

- 多角色与细粒度权限配置；
- 审批、多人协作、任务分派和消息催办；
- 修改历史积分流水、核销记录或活动历史；
- 监管端编辑服务商品质、对象、区域或活动；
- 独立数据仓库、复杂预测算法和自动化运营策略；
- 真实接口、跨端同步机制与生产数据迁移实现。

## 6. 信息架构

一期有四个一级页面：

```text
运营驾驶舱
对象档案
积分管理
活动管理
```

对象档案内包含四个对象工作区：犬只与犬主人、城管队员、服务商、地图区域。积分仅在对象详情中展示摘要，所有积分修改均进入积分管理页面。

## 7. 运营驾驶舱

### 7.1 页面目标

帮助超级管理员在进入系统后快速回答：哪些端口与对象数据发生变化、设备是否健康、积分是否按预期流转、服务商与活动是否有效，以及需要优先关注什么。

### 7.2 首屏内容

- 数据更新时间与当前统计周期；
- 各端活跃概览：犬主人端、城管端、服务端、监管端；
- 对象增长与设备健康：在册犬只、电子狗牌数量、设备在线 / 异常概况；
- 积分经营概览：本周期发放、核销、余额、使用率；
- 服务商与活动概览：有效服务商数、核销表现、进行中活动数、活动表现；
- 待关注异常占位：设备异常、积分异常、服务商低表现、活动临近或已下线提示。

### 7.3 运营洞察扩展

驾驶舱新增三项聚合洞察，用于建立“行为热度 → 咨询需求 → 兑换需求”的运营判断链路：

- 城市遛狗热力图：按区域和时段汇总电子狗牌定位数据，展示遛狗活跃热度；支持“近 7 日、早间、晚间”等时段切换；
- AI 咨询问题 Top10：展示咨询问题类别、咨询量、占比与变化，用于识别犬主高频问题；
- 宠物服务需求 Top10：展示积分兑换关联的服务或商品类别、兑换量、占比与变化，用于优化服务商供给与活动策划。

热力图和排行榜均只展示聚合结果：不展示单犬实时位置、轨迹、犬主人身份、AI 对话正文或未脱敏内容。热力图仅使用电子狗牌定位数据，不接入犬主手机定位和普通狗牌数据。

### 7.4 AI 运营分析

驾驶舱底部提供全宽 AI 运营分析区，基于当前驾驶舱的聚合数据输出三项内容：当前运营重点、建议动作和对应数据依据摘要。

AI 不展示个体定位、犬主身份或咨询对话正文，不自动修改对象、积分、服务商、区域或活动数据，也不自动创建任务。

一期使用可信演示数据与清晰口径验证驾驶舱，不承诺真实数据接入。具体指标名称、阈值和图表形态可在后续评审中迭代，但不得新增未在本 PRD 确认的业务模块。

### 7.3 下钻

驾驶舱中的对象、积分、服务商与活动摘要可跳转至相应一级页面并带入基础筛选条件。驾驶舱不直接提供编辑、删除、积分调整或活动上下线操作。

## 8. 对象档案

### 8.1 通用规则

- 默认展示有效对象，支持按状态查看已删除 / 已下线对象；
- 提供列表、关键词查询、详情、新增、编辑、逻辑删除 / 下线与恢复；
- 删除、下线和恢复均需二次确认并填写原因；
- 积分字段在此只读展示；
- 页面右下角固定提供“操作历史”悬浮入口，点击后打开与当前对象类型或当前对象关联的历史列表；
- 操作历史不允许编辑或删除。

### 8.2 犬只与犬主人

犬只记录以犬只为主对象，可关联一个犬主人手机号。字段如下：

| 字段 | 规则 |
| --- | --- |
| 犬只姓名 | 必填 |
| 品种 | 必填 |
| 年龄 | 必填，非负数 |
| 性别 | 必填 |
| 犬只 ID | 系统唯一，不允许与现有记录重复 |
| 狗牌编号 | 必填，系统内唯一 |
| 狗牌种类 | 必填，普通狗牌或电子狗牌 |
| 内部设备编号 | 仅电子狗牌必填；普通狗牌为空 |
| 犬主人手机号 | 必填，按手机号格式校验 |
| 当前积分 | 只读摘要，跳转积分管理查看流水 |
| 状态 | 有效、已删除 |

### 8.3 城管队员

| 字段 | 规则 |
| --- | --- |
| 姓名 | 必填 |
| 队员编号 | 必填且唯一 |
| 所属大队 | 必填 |
| 辖区 | 必填，可关联已维护的区域 |
| 核销犬只次数 | 只读累计值，不允许手工改写历史核销 |
| 状态 | 有效、已删除 |

### 8.4 服务商

| 字段 | 规则 |
| --- | --- |
| 服务商名称 | 必填 |
| 地址 | 必填 |
| 服务商类别 | 必填 |
| 服务商品质 | 必填，由运营端维护；监管端只读 |
| 积分核销数 | 只读累计值 |
| 关联活动 | 只读摘要，可跳转活动管理 |
| 状态 | 有效、已删除 |

服务商品质的具体评级维度、等级名称与阈值在一期原型中采用固定演示口径；若后续形成正式评级规则，应另立决策并同步 PRD。

### 8.5 地图区域

区域工作区管理犬只禁入区域与宠物友好区域。

| 字段 | 规则 |
| --- | --- |
| 区域名称 | 必填 |
| 区域类型 | 必填：犬只禁入区域、宠物友好区域 |
| 所属街道 / 社区 | 必填 |
| 位置与边界描述 | 必填；一期原型可用文字与演示范围表达 |
| 关联活动 | 仅友好区域展示，只读摘要 |
| 状态 | 已上线、已下线 |

区域可新建、编辑、下线和重新上线。下线区域不再向其他端口的默认有效数据提供展示。

## 9. 积分管理

### 9.1 页面目标

积分管理是所有积分解释和运营操作的唯一入口。对象档案不得绕过本页修改积分。

### 9.2 积分规则

支持新增、编辑、启用、停用积分获取与积分核销规则。每条规则至少包括：

- 规则名称与规则编号；
- 规则类型：获取或核销；
- 适用对象；
- 触发条件或使用场景；
- 积分数值；
- 生效时间与状态；
- 规则说明。

规则发布、修改、启停均需记录操作历史。已用于生成历史流水的规则可停用，但不得通过修改规则反向改写既有流水。

### 9.3 积分流水查询

支持按对象查询积分变动历史，并至少展示：流水编号、变动时间、关联对象、变动类型、变动积分、变动后余额、来源规则或人工调整原因、操作人。

积分流水仅可查询，禁止编辑、删除和回写。

### 9.4 人工积分调整

超级管理员可为指定对象发放或扣减积分。提交前必须填写调整原因，系统生成一条新的不可修改流水，记录调整前余额、调整值和调整后余额。

人工调整不能修改或覆盖已有流水，也不能在对象档案页直接执行。

### 9.5 经营分析

展示当前周期的积分发放、使用 / 核销、余额及使用率，并支持按对象类型、服务商和时间进行基础筛选。使用率定义为：当前周期已核销积分 ÷ 当前周期已发放积分；当分母为零时明确展示暂无可计算数据。

### 9.6 操作历史

页面右下角固定提供“操作历史”悬浮入口。至少记录规则发布、规则修改、规则启停和人工积分调整；人工调整记录必须带原因。

## 10. 活动管理

### 10.1 页面目标

运营端是活动的创建和维护端；监管端仅查看运营端已上线的活动信息。

### 10.2 活动字段

| 字段 | 规则 |
| --- | --- |
| 活动名称 | 必填 |
| 活动状态 | 草稿、已上线、已下线、已结束 |
| 活动时间 | 必填，含开始与结束时间 |
| 举办地点 | 必填，可关联服务商、宠物友好区域或其他已确认场地 |
| 关联服务商 | 可选 |
| 关联宠物友好区域 | 可选 |
| 活动说明 | 必填 |
| 参与提示 | 可选 |
| 创建 / 更新时间 | 系统记录 |

### 10.3 操作与状态

- 支持新建、编辑、上线、下线、结束、查看详情和查看操作历史；
- 草稿不可被其他端口读取；
- 已上线活动可被监管端只读展示；
- 下线和结束活动保留历史，但不在其他端口默认活动列表中展示；
- 活动时间结束后可由超级管理员标记为已结束；
- 一期不建设报名、参与人管理、签到或活动效果自动计算。

## 11. 操作留痕

### 11.1 记录范围

至少记录对象新增、编辑、删除、恢复，服务商品质修改，区域与活动上线 / 下线，积分规则发布 / 修改 / 启停及人工积分调整。

### 11.2 记录字段

每条记录至少包含：操作时间、操作者、模块、操作对象、操作类型、变更前值、变更后值和操作原因。没有前值或后值的创建、删除操作应明确标记为空。

### 11.3 查询与安全

操作历史只读、不可修改、不可删除。对象档案与积分管理必须提供右下角悬浮入口；活动与区域的操作历史可从详情页查看。

## 12. 关键用户路径

### 12.1 维护服务商品质

```text
进入对象档案
→ 选择服务商
→ 查询并进入服务商详情
→ 编辑服务商品质
→ 保存
→ 系统记录前后值与原因
→ 监管端在读取有效数据时展示最新品质
```

### 12.2 人工调整积分

```text
进入积分管理
→ 查询目标对象
→ 发起积分发放或扣减
→ 填写调整原因
→ 提交
→ 生成不可修改的新增流水
→ 可在流水与操作历史中追溯
```

### 12.3 下线活动

```text
进入活动管理
→ 选择活动
→ 执行下线并填写原因
→ 活动不再在其他端默认列表中展示
→ 历史活动、关联对象和操作记录仍可查询
```

## 13. 原型与实现约束

- 新建独立运营端 HTML 原型与资产路径，使用 `05_` 前缀；
- 原型仅覆盖本 PRD 确认的一期范围，不私自扩展审批、多人角色、报名或任务等模块；
- 采用可信且相互关联的演示数据，确保犬只、狗牌、设备、积分、服务商、区域与活动的关系一致；
- “删除”按钮、状态筛选、下线与恢复状态必须在原型中可验证；
- 操作历史入口与人工积分调整原因必填必须在原型中可验证；
- 原型不接入真实业务数据或真实地图服务。

## 14. 验收标准

### 14.1 产品结构

1. 有且仅有运营驾驶舱、对象档案、积分管理、活动管理四个一级页面。
2. 一期默认只以超级管理员身份展示，无低级角色配置入口。
3. 驾驶舱只用于查看与下钻，不直接编辑业务数据。
4. 驾驶舱展示遛狗热力图、AI 咨询问题 Top10 和宠物服务需求 Top10；三者均为聚合洞察，不暴露个体定位、身份或对话内容。
5. 驾驶舱在 1920×1080 下满足紧凑四 KPI、1.45:1:1 洞察行、四等分运营行和全宽 AI 分析区的固定模块尺寸；模块内部内容不得溢出。
6. AI 运营分析只展示聚合重点、建议和依据摘要，不具备自动执行能力。

### 14.2 对象与区域

1. 四类对象工作区均可查询、查看详情、新增、编辑、逻辑删除 / 下线、恢复及按状态筛选。
2. 犬只字段、城管字段、服务商字段与区域字段满足本 PRD 第 8 章要求。
3. 对象档案中的积分仅为只读摘要，无法直接修改。
4. 电子狗牌的内部设备编号只在电子狗牌类型下要求填写。
5. 服务商品质可在运营端修改，监管端无编辑入口。

### 14.3 积分

1. 积分规则可维护并有状态与生效时间。
2. 积分流水可按对象查询，且无编辑、删除入口。
3. 人工调整必须填写原因并新增流水，不能覆盖历史流水。
4. 可查看发放、核销、余额与使用率，并对零分母给出明确空状态。

### 14.4 活动与留痕

1. 活动可创建、编辑、上线、下线和结束；监管端仅读取已上线活动。
2. 删除 / 下线不物理清除对象及其历史关联数据。
3. 对象档案和积分管理的右下角均有操作历史悬浮入口。
4. 操作历史至少显示操作人、时间、对象、前后值与原因，且不可编辑或删除。

```text
运营后台端一期是超级管理员统一维护跨端业务数据、运营积分和活动，并用完整留痕保障可解释性的工作后台。
```
