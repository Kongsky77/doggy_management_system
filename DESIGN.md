# 监管后台端 DESIGN

本文件用于沉淀监管后台端当前可直接执行的视觉与排版规则。它不是业务 PRD 的替代物，而是把已确认的设计方向转译为可落地的画布、布局、组件和扩展规范，供后续原型、Figma 复刻、HTML 原型和前端实现共同遵循。

当前适用对象：

- `01_项目PRD及开发/02_监管后台端一期PRD_v1.1.md`
- `01_项目PRD及开发/02_监管后台端一期原型_v1.0.html`

## 1. 设计目标

监管后台端一期不是展示型大屏，也不是复杂业务中台，而是面向监管人员的高频 PC 工作台。页面设计应优先满足：

1. 首屏快速扫读
2. 地图、风险、趋势和列表并行浏览
3. 在紧凑信息密度下保持清晰层级
4. 在后续扩展指标或卡片时不破坏整体秩序

## 2. 总体风格

- 品牌主色：CDHT 橙色体系
- 页面气质：专业、克制、温和、可信
- 背景基调：浅灰蓝工作台背景，不用纯白满屏
- 卡片气质：白底、细边框、弱阴影、弱圆角
- 状态色只作语义色，不作全局主色

推荐基础色：

- `--bg-page: #F7F9FC`
- `--bg-card: #FFFFFF`
- `--border-soft: #EEF1F5`
- `--text-primary: #252A32`
- `--text-secondary: #6B7280`
- `--brand-orange: #FF8F1F`
- `--success: #34C759`
- `--warning: #FFB020`
- `--danger: #FF5A4F`
- `--info: #3B82F6`

## 3. 画布规则

当前监管后台端驾驶舱默认使用：

- 画布尺寸：`1920 x 1024`

该尺寸不是对旧版 `1536 x 1024` 的简单拉伸，而是重排版。处理原则：

1. 页面高度节奏基本稳定
2. 横向新增空间优先分配给地图区、事件区和四列信息卡
3. 左侧导航仅小幅增宽，不做夸张扩展
4. KPI 区按等分重排，不靠压缩间距硬塞

## 4. 页面骨架

### 4.1 一级布局

- 左侧导航：`280 x 1024`
- 右侧主区：`1640 x 1024`
- 顶部栏：`1640 x 72`
- 主内容起点：`x=304, y=96`
- 全局内容边距：`24`
- 模块标准间距：`16`

### 4.2 模块分区

推荐高度分配：

- KPI 区：`112`
- 上半区主内容：`404`
- 趋势区：`204`
- 底部区：`160`

推荐模块尺寸：

- KPI 卡：`315 x 112`，5 张时单排
- 地图卡：`1036 x 404`
- 实时事件卡：`588 x 404`
- 趋势卡：`398 x 204`，4 张单排
- 底部卡：`398 x 160`，4 张单排

## 5. 组件规则

### 5.1 顶部栏

- 保持单行工具栏结构
- 左侧只保留：折叠菜单、页面标题
- 右侧保留：搜索、通知、全屏、账号
- 不在顶部长期堆放解释性文案

### 5.2 左侧导航

- 导航宽度：`280`
- 一级项使用单行标题
- 高亮项使用浅橙底 + 左侧品牌色竖条
- 图标保持轻量，不用重块状导航

### 5.3 卡片系统

- 卡片圆角：`18-20`
- 卡片边框：`1px solid #EEF1F5`
- 卡片阴影：低透明度暖色弱阴影
- 标题字号：`14-16`
- 说明文字字号：`12-13`

### 5.4 KPI 卡片

默认规则：

- KPI 区优先单排展示
- 单卡信息结构为：图标 + 中文短标题 + 主数值 + 短变化值
- 不放英文标题
- 不放长说明文案

5 张 KPI 时：

- 单卡尺寸：`315 x 112`

6 张 KPI 时：

- 单卡尺寸：`260 x 112`
- 卡间距仍为 `16`
- 图标容器从 `52 x 52` 收紧到 `44 x 44`
- 主数值字号从 `24` 收紧到 `20-22`
- 文案整体缩一档，但仍保持单排

不推荐：

- 把 KPI 拆成两行 `3 + 3`
- 把卡间距压到 `8`
- 保持 5 卡宽度不变后强塞第 6 张

### 5.5 地图卡

- 标题栏高度：`52`
- 左侧保留图例与区域数量
- 地图热点按“中心 + 四周分布”重排
- 地图控件右下固定
- 不做深色指挥大屏风格地图

### 5.6 实时事件卡

- 标题栏高度：`52`
- 每条事件高度：`64-68`
- 单条结构：事件图标、标题、摘要、时间、状态标签
- 优先展示可落到一期数据字段的事件

### 5.7 趋势卡

- 4 张单排等宽
- 顶部标题 + 时间范围切换
- 中部主值 + 周同比
- 底部橙色折线图

### 5.8 底部卡

底部 4 卡默认承载：

1. 犬只状态统计
2. 设备状态统计
3. 区域风险等级
4. 快捷操作

如需替换内容，仍应保持四列等宽结构，不单独做异形卡。

## 6. 字体与文案

- 标题：`SemiBold`
- 主值：`SemiBold`
- 正文：`Regular`
- 所有 KPI、趋势和列表文案优先中文短语化
- 指标文案优先使用“主数值 + 短变化值”的表达

示例：

- `在册犬只总数`
- `电子狗牌在线率`
- `今日异常事件`
- `待处理任务`
- `积分参与率`
- `今日扫码核验`

## 7. 页面扩展规则

### 7.1 适合扩展的内容

- KPI 指标数量扩展
- 事件列表字段微调
- 趋势图指标替换
- 快捷操作项替换

### 7.2 不应轻易打破的结构

- 左导航 + 顶部栏 + 主区的骨架
- 5/6 KPI 单排规则
- 上半区地图左、事件右的双栏结构
- 趋势 4 卡单排
- 底部 4 卡单排

### 7.3 新增指标的优先级

如 KPI 新增一张，优先考虑：

1. 今日扫码核验
2. 任务办结率
3. 高风险区域数
4. 设备低电量数
5. 在线巡查人员数

其中“今日扫码核验”最符合一期锚点。

## 8. Figma 与 HTML 执行建议

### 8.1 Figma

- 先搭骨架，再填内容
- 先确定模块尺寸，再补图标和图表
- 命名建议使用：`Page / Region / Card / Item`

### 8.2 HTML 原型

- 用统一 CSS 变量维护颜色、圆角、边框和阴影
- 保持桌面端 `1920 x 1024` 的首屏节奏
- 不为了适配演示而破坏监管后台的一致密度

## 9. 验收检查表

- 首屏是否能同时看到 KPI、地图、事件、趋势和底部摘要
- KPI 是否保持中文短标题与紧凑表达
- 6 张 KPI 时是否仍为单排
- 左导航是否轻量、不过宽
- 地图卡与事件卡是否高度一致
- 趋势卡与底部卡是否保持四列等宽
- 页面是否仍呈现“监管工作台”而不是“营销大屏”
