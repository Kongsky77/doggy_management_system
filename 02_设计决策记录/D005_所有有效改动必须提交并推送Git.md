# D005 所有有效改动必须提交并推送 Git

## 1. 背景与触发原因

项目持续积累 PRD、原型、设计决策、Skill 和方法论等多类资产。如果改动只保留在本地工作区，后续会出现以下问题：

- 版本边界不清
- 历史回溯困难
- 新会话或协作者无法准确接手
- 某些关键方案和规则容易丢失

因此需要把 Git 提交与推送从“建议动作”升级为明确铁则。

---

## 2. 本次决策内容

将以下规则作为项目强制协作约束：

1. 只要发生有效改动，无论改动对象是文档、PRD、原型、决策记录、Skill 还是方法论，都必须进入 Git 历史
2. 每轮有意义的改动都必须及时 `git commit`
3. 每次提交后都必须尽快 `git push` 到 GitHub
4. 版本留痕属于工作结果的一部分，不能视作可选收尾动作

---

## 3. 关键取舍

### 方案 A：仅在大阶段完成后再统一提交 / 推送

优点：

- 操作次数少

缺点：

- 版本粒度过粗
- 中间状态容易丢失
- 回溯成本高
- 协作接力风险大

### 方案 B：每轮有效改动都提交并尽快推送

优点：

- 版本清晰
- 回溯方便
- 协作稳定
- 文档、原型和决策可以同步留痕

缺点：

- 需要更严格地管理提交粒度

---

## 4. 最终决策

采用方案 B。

Git 提交和推送不再是“最好这样做”，而是项目级强制规则。

---

## 5. 影响范围

本次决策影响：

- `AGENTS.md`
- 后续所有 Codex / 人工协作流程
- PRD、原型、设计决策、Skill、方法论的提交节奏
