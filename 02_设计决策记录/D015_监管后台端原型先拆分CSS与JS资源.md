# D015 监管后台端原型先拆分 CSS 与 JS 资源

## 背景

监管后台端一期原型持续迭代后，单个 HTML 文件已经同时承载页面结构、样式系统、mock 数据和交互逻辑。随着第六步“页面跳转和模拟数据联动”即将开始，继续在单文件内修改会带来以下问题：

1. 每次改动都需要读取大量无关 CSS、HTML 和 JS
2. 后续交互开发更容易误伤已稳定的视觉样式
3. mock 数据和事件绑定混在页面结构中，定位成本较高
4. 单文件行数过多，会降低后续 AI 协作和人工维护效率

## 决策

监管后台端一期原型先采用资源拆分方案：

```text
01_项目PRD及开发/
  02_监管后台端一期原型_v1.0.html
  02_监管后台端一期原型_assets/
    styles.css
    app.js
```

其中：

1. HTML 保留页面结构和资源引用
2. CSS 统一放入 `styles.css`
3. mock 数据、页面切换、抽屉、地图点位和对象档案渲染逻辑统一放入 `app.js`
4. 暂不按页面拆分 HTML，确保原型仍可直接通过本地 `file://` 打开

## 未采用方案

### 立即按页面拆分 HTML

暂不采用。

原因是按页面拆分后，普通本地文件打开方式会遇到 HTML 片段加载和跨文件引用复杂度问题。如果引入本地服务或构建流程，会增加当前原型阶段的使用门槛。

### 继续保持单文件原型

不再采用。

原因是第六步交互联动会继续增加 JS 和 mock 数据，继续堆在单文件中会显著降低后续维护效率。

## 原型影响

- `01_项目PRD及开发/02_监管后台端一期原型_v1.0.html`
- `01_项目PRD及开发/02_监管后台端一期原型_assets/styles.css`
- `01_项目PRD及开发/02_监管后台端一期原型_assets/app.js`

本次调整不改变 UI、不改变业务内容、不新增一期功能，只改变原型代码组织方式。

后续第六步交互开发应优先在 `app.js` 中推进；视觉样式调整优先在 `styles.css` 中推进；HTML 仅在页面结构确实变化时修改。
