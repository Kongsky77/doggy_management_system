# D006 前期 main 推进，后期按端口分支开发

## 1. 背景与触发原因

当前仓库会持续承接多个端口的产品与原型开发工作。在项目早期，多个端口都还处于快速成型阶段，直接在 `main` 上推进效率较高；但当各端口一期逐步完成后，如果仍长期把所有后续改动都直接提交到 `main`，会带来以下问题：

- 各端口后续演进历史互相掺杂
- 回溯某一端口版本变化的成本变高
- 根目录规则改动和端口业务改动容易绑定在同一条主线上
- `main` 难以保持稳定主线语义

因此需要定义一套分阶段的 Git 协作策略。

---

## 2. 本次决策内容

采用“前期以 `main` 为主，后期按端口分支开发”的版本管理策略。

具体规则如下：

1. 在多个端口都处于一期建设阶段时，可以直接在 `main` 上推进
2. 这一阶段每次提交都应尽量保持单一主题，不要混入多个端口的无关改动
3. 当所有端口的一期开发完成后，涉及某个端口的后续迭代，应在对应端口分支上进行
4. 端口分支完成验证后，再合并回 `main`
5. 根目录协作规则、目录结构、全局管理约定等纯全局改动，可以直接进入 `main`
6. 如果根目录改动本质上由某个端口需求触发，实践上优先在该端口分支完成后再合回 `main`

---

## 3. 关键取舍

### 方案 A：始终所有改动都直接提交到 `main`

优点：

- 简单直接
- 前期推进速度快

缺点：

- 后期历史会越来越杂
- 不利于按端口追踪改动
- 稳定主线和实验性改动边界不清

### 方案 B：从一开始所有端口都严格走分支

优点：

- 结构清晰
- 分支边界明确

缺点：

- 对当前仍在快速成型阶段的项目来说管理成本偏高
- 前期频繁切分支会降低推进效率

### 方案 C：前期 `main` 推进，后期按端口分支开发

优点：

- 兼顾前期效率与后期可维护性
- 与项目当前阶段更匹配
- 能逐步把 `main` 过渡成稳定主线

缺点：

- 需要团队对“何时进入后期策略”有共同认知

---

## 4. 最终决策

采用方案 C。

版本管理按项目阶段执行：

```text
多端口一期建设阶段：以 main 为主
所有端口一期完成后：按端口分支开发，main 保持稳定主线
```

建议的后期分支命名方式可以采用：

```text
feature/regulatory-v1.1
feature/city-inspection-v1.1
feature/owner-app-v1.1
```

---

## 5. 影响范围

本次决策影响：

- `AGENTS.md`
- 后续所有 Git 协作习惯
- 各端口二期或后续版本的迭代方式

---

## 6. 执行提示

1. 当前阶段仍可继续使用 `main`
2. 一旦进入“所有端口一期完成”的节点，应把该状态明确记录下来
3. 从那之后，端口需求默认走各自分支，根目录仅保留纯全局改动直改 `main`
