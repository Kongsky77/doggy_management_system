# 项目协作规则

本文件是给后续 Codex / AI 编码会话读取的项目工作协议。新会话开始前，应先阅读本文件，再阅读根目录 `README.md` 以及 00-04 文件夹内的 `README.md`。

## 项目定位

本仓库用于沉淀犬只管理系统相关的 PRD、原型、设计决策、复用 Skill 和最终方法论。

当前项目资产不是普通散文件，而是按“辅助资料 -> 需求定义 -> 决策留痕 -> 能力复用 -> 方法沉淀”的方式管理。

## 目录规则

- `00_辅助资料/`：存放设计方案、概念图、交接稿、讨论稿和其他辅助理解资料。这些资料有参考价值，但不是当前开发与原型的直接执行依据。
- `01_项目PRD及开发/`：存放当前有效 PRD、功能说明，以及依据 PRD 制作的原型 HTML。只有本目录中的 PRD 和原型文件可作为当前需求与开发的直接依据。
- `02_设计决策记录/`：存放开发过程中发生的需求变更、变更原因、方案取舍和影响范围。
- `03_skill蒸馏/`：存放开发过程中提炼出的可复用 Skill、规则和工作模板。
- `04_最终方法论/`：存放经过项目验证后，可以稳定复用的方法论。

## 文件命名规则

- 业务文件名前缀中的 `01_`、`02_`、`03_`、`04_` 等编号，表示“端口编号”或“业务对象编号”，不是文档顺序号。
- 例如当前监管后台端作为第二个端口，其相关文件应统一使用 `02_` 作为业务前缀。
- 同一份文档后续迭代时，不通过改动前缀编号区分版本，而是直接更新文件名尾部版本号，例如 `v1.0`、`v1.1`、`v1.2`。
- 不要把同一端口的不同文档误命名成 `02_`、`03_`、`04_` 这种连续递增格式；如果它们都属于同一个端口，应保持同一个业务前缀。
- 文档命名建议采用：`端口编号_业务对象_文档类型_v版本号.md`。

## 工作流规则

1. 开始任何新任务前，可先从 `00_辅助资料/` 获取背景，但必须确认当前需求依据来自 `01_项目PRD及开发/`。
2. 如果用户提出需求变更，应同步更新 `02_设计决策记录/`，说明变更背景、决策、未采用方案和影响范围。
3. 如果需求变更会影响 PRD 或原型，应同时更新 `01_项目PRD及开发/` 中对应文件，保持 PRD、原型和决策记录一致。
4. 如果开发过程中出现可复用经验，应沉淀到 `03_skill蒸馏/`。
5. 只有经过项目验证、适合跨项目复用的内容，才进入 `04_最终方法论/`。

## 版本管理规则

- Git 版本管理是本仓库的铁则。无论是文档、PRD、原型、决策记录、Skill 还是方法论，只要发生有效改动，都必须进入 Git 历史。
- 每轮有意义的改动都必须 `git commit`，不得长期停留在未提交状态。
- 每次提交后都必须尽快 `git push` 到 GitHub，保证远端有可回溯版本。
- 不允许把“改完但还没来得及提交/推送”当成常态；版本留痕是工作结果的一部分，不是收尾时可选动作。
- 当前处于多端口持续建设阶段时，可以直接在 `main` 上推进，但每次提交都应尽量保持单一主题，不要把多个端口的无关改动混入同一提交。
- 当所有端口的一期开发完成后，后续涉及某个端口的改动，应优先在对应端口分支上开发，完成后再合并回 `main`。
- 根目录规则、目录结构、协作协议等纯全局性改动，可以直接在 `main` 上进行；如果根目录改动本质上是由某个端口需求触发，也应优先在该端口分支完成后再合回 `main`。
- 提交信息应准确表达本轮目的，例如 `feat: add task evidence capture` 或 `docs: record scope decision`。
- 不要把无关文件混进同一次提交；如果发现未跟踪文件与当前任务无关，应保留但不提交。
- 不要重置或回滚用户已有改动，除非用户明确要求。

## 原型与 PRD 同步规则

- 原型 HTML 只能表达 PRD 中已确定的一期范围，不应私自扩展成新模块。
- 如果原型新增入口、按钮、页面或流程，PRD 中必须能找到对应说明。
- 如果 PRD 删除或后置某功能，原型中也应移除或降级该功能入口。
- 设计决策记录解释“为什么这样改”，PRD 描述“当前要做什么”，原型展示“用户如何使用”。
- `00_辅助资料/` 中的设计方案、概念图和交接材料不能直接替代 PRD；如果其中结论被采纳，必须同步进入 `01_项目PRD及开发/` 或 `02_设计决策记录/`。

## 当前一期产品边界

一线巡查端一期聚焦“查犬 + 看位置 + 处理任务”，以扫码核验作为主要锚点。

当前一期不做完整巡查地图、复杂权限、轨迹回放、AI 异常识别、任务转派、多人协同和独立取证材料库。

“现场取证”仅作为任务详情页处理区中的轻量附件能力，不作为独立一级功能。

## 新会话启动建议

如果这是一个新的 Codex 会话，建议先执行以下动作：

1. 阅读 `AGENTS.md`。
2. 阅读根目录 `README.md`。
3. 阅读 00-04 文件夹内的 `README.md`。
4. 查看 `git status --short --branch`，确认当前工作区状态。
5. 根据用户当前任务，只修改相关文件，并在完成后提交、推送。
