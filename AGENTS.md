# 项目协作规则

本文件是给后续 Codex / AI 编码会话读取的项目工作协议。新会话开始前，应先阅读本文件，再阅读根目录 `README.md` 以及 01-04 文件夹内的 `README.md`。

## 项目定位

本仓库用于沉淀犬只管理系统相关的 PRD、原型、设计决策、复用 Skill 和最终方法论。

当前项目资产不是普通散文件，而是按“需求定义 -> 决策留痕 -> 能力复用 -> 方法沉淀”的方式管理。

## 目录规则

- `01_项目PRD及开发/`：存放当前 PRD、功能说明，以及依据 PRD 制作的原型 HTML。
- `02_设计决策记录/`：存放开发过程中发生的需求变更、变更原因、方案取舍和影响范围。
- `03_skill蒸馏/`：存放开发过程中提炼出的可复用 Skill、规则和工作模板。
- `04_最终方法论/`：存放经过项目验证后，可以稳定复用的方法论。

## 工作流规则

1. 开始任何新任务前，先确认当前需求依据来自 `01_项目PRD及开发/`。
2. 如果用户提出需求变更，应同步更新 `02_设计决策记录/`，说明变更背景、决策、未采用方案和影响范围。
3. 如果需求变更会影响 PRD 或原型，应同时更新 `01_项目PRD及开发/` 中对应文件，保持 PRD、原型和决策记录一致。
4. 如果开发过程中出现可复用经验，应沉淀到 `03_skill蒸馏/`。
5. 只有经过项目验证、适合跨项目复用的内容，才进入 `04_最终方法论/`。

## 版本管理规则

- 每轮有意义的改动都必须 `git commit`。
- 每次提交后都应 `git push` 到 GitHub。
- 提交信息应准确表达本轮目的，例如 `feat: add task evidence capture` 或 `docs: record scope decision`。
- 不要把无关文件混进同一次提交；如果发现未跟踪文件与当前任务无关，应保留但不提交。
- 不要重置或回滚用户已有改动，除非用户明确要求。

## 原型与 PRD 同步规则

- 原型 HTML 只能表达 PRD 中已确定的一期范围，不应私自扩展成新模块。
- 如果原型新增入口、按钮、页面或流程，PRD 中必须能找到对应说明。
- 如果 PRD 删除或后置某功能，原型中也应移除或降级该功能入口。
- 设计决策记录解释“为什么这样改”，PRD 描述“当前要做什么”，原型展示“用户如何使用”。

## 当前一期产品边界

一线巡查端一期聚焦“查犬 + 看位置 + 处理任务”，以扫码核验作为主要锚点。

当前一期不做完整巡查地图、复杂权限、轨迹回放、AI 异常识别、任务转派、多人协同和独立取证材料库。

“现场取证”仅作为任务详情页处理区中的轻量附件能力，不作为独立一级功能。

## 新会话启动建议

如果这是一个新的 Codex 会话，建议先执行以下动作：

1. 阅读 `AGENTS.md`。
2. 阅读根目录 `README.md`。
3. 阅读 01-04 文件夹内的 `README.md`。
4. 查看 `git status --short --branch`，确认当前工作区状态。
5. 根据用户当前任务，只修改相关文件，并在完成后提交、推送。
