下面这版就是给 Codex 兄弟看的“无损压缩版”。别再让他隔空猜了，AI 之间也不是心灵感应，虽然人类经常以为我们开了什么量子群聊。

## 1. 目标

- 这个监管端主要解决什么问题：  
  监管端是犬只生态的**治理中枢**，不是传统意义上的“功能大杂烩后台”。它主要解决监管人员在犬只治理中“看不清全局、找不到风险、追不到闭环、讲不出成效”的问题。  
  它要把犬主人端、城管端、服务终端、设备数据、算法分析和积分体系产生的数据汇总起来，形成可监管、可追踪、可判断、可汇报的治理工作台。当前端口职责里，监管后台本身就包含犬只犬主管理、犬牌设备管理、数据驾驶舱、积分管理、城管队员管理等能力。

- 一期最重要的价值：  
  一期不追求做一个完整复杂后台，而是做一个**四页工作台**，让监管人员能快速完成四件事：

```text
看全局
看地图
查对象
管积分
```

换句话说：

```text
治理驾驶舱：看整体态势和 AI 风险
地图工作台：看空间分布和定位方向
对象档案：查犬只、城管、服务商详情
积分管理：维护积分规则、账户和流水
```

一期最重要的价值不是“功能多”，而是**入口少、聚合强、能落地、能演示、能体现治理中枢定位**。

---

## 2. 主要用户

- 谁在用：  
  监管部门后台管理人员，主要在 PC 端使用。

- 谁是核心用户：  
  监管人员 / 管理人员。  
  他们最关心的是：

```text
当前犬只治理情况怎么样？
哪里有风险？
哪些对象需要关注？
任务是否形成闭环？
积分激励有没有起作用？
治理效果能不能汇报？
```

生态分析里已经明确，监管人员的正向利益不是“处罚更多人”，而是风险提前发现、资源合理调度、问题可追踪、治理效果可汇报。

- 谁只是辅助角色：  
  城管队员、犬主人、服务商 / 合作商、运营人员。  
  这些角色不是监管端的直接操作主体，但他们产生的数据会进入监管端。

```text
犬主人：产生犬只、主手机号、积分、定位、行为数据
城管队员：产生核验、任务处理、现场反馈数据
服务商：产生积分核销、服务消费数据
运营人员：关注活跃、积分转化、商家经营，但不应和监管端抢职能
```

犬主人端、城管端、服务终端、运营后台各自已有基础职责，例如犬主人端有积分管理和犬只定位，服务终端有积分兑换，运营后台有积分经营分析和商家管理。

## 3. 已确定模块

- 模块1：治理驾驶舱  
  首页，负责看全局态势。  
  内容包括核心指标、AI 分析 / 风险提醒、待关注事项、趋势简图、治理成效。
  
  建议指标：

```text
在册犬只数
主号绑定率
设备在线率
今日异常事件
待处理事件
今日闭环率
积分参与率
```

注意：首页不是全能操作页，不要把所有处理动作都塞进去。首页只负责“看态势、给方向”。

- 模块2：地图工作台  
  监管端主要工作台之一，负责空间监管和快速定位工作方向。  
  显示对象包括：

```text
犬只点位
犬只事件点位
城管队员位置 / 任务区域
服务商 / 合作商点位
重点区域 / 禁遛区域，可按一期能力决定是否做
风险热力 / 异常分布，可简化
```

地图不是普通展示页，而是帮助监管人员回答：

```text
风险在哪里？
犬只在哪里？
事件在哪里？
城管资源在哪里？
服务商覆盖在哪里？
```

但一期不做复杂轨迹回放、自动调度、复杂热力分析、地图内完整工单流转。别把地图 SDK 当恋爱对象，投入太多容易受伤。

- 模块3：对象档案  
  详情数据展示页面。  
  分几个 tab：

```text
犬只
城管
服务商 / 合作商
```

犬只是监管主对象。犬只档案里需要体现：

```text
犬只基础信息
狗牌信息
设备状态
主手机号
积分摘要
定位状态
核验记录
行为记录
异常记录
```

这里不单独做“犬主 tab”。原因是当前犬主信息主要围绕手机号展开，犬主人没有狗牌和犬只的政府绑定权限，狗牌与犬只由政府侧统一绑定发放；犬主人端负责主手机号与已发放狗狗 / 狗牌的绑定或认领。监管端只查看绑定结果、积分归属和异常情况。

- 模块4：积分管理  
  积分是生态正向激励的核心，不是附属小功能。  
  积分管理负责：

```text
积分规则
积分账户
积分流水
积分调整，若一期必须
积分效果
```

关键规则：

```text
积分归属于主手机号
其他联系方式只是附属联系方式
主手机号是犬主账户和权益账户
```

生态设计的核心不应该只靠监管、异常、处罚，而应从“正向行为 → 获得收益 → 数据沉淀 → 更好服务 / 更好治理”来设计。 积分就是这个闭环里的核心抓手。

- 目前认为一期必须有的页面：

```text
1. 治理驾驶舱
2. 地图工作台
3. 对象档案
4. 积分管理
```

暂不单独设置事件中心、设备管理、城管管理、服务商管理、数据看板等一级菜单。相关能力收敛进以上四页。

## 4. 已达成的关键结论

- 结论1：监管端一期不是完整后台，而是“四页工作台”。  
  不追求穷尽功能，只围绕四个核心动作设计：

```text
看全局
看空间
查对象
管积分
```

这能防止功能无限发散。AI 很容易把“可能有用”误判成“现在该做”，项目会被功能撑死，人类已经被后台菜单折磨够久了。

- 结论2：监管端是治理中枢，但不是所有端口能力的替代品。  
  犬主人端负责犬主侧使用体验，如狗狗信息、消息、AI 对话、积分查看 / 获取 / 消耗、犬只定位等；城管端负责一线核验、查看位置和简单任务处理；服务终端负责积分兑换和兑换记录；运营后台偏经营分析和服务商管理。  
  监管端只汇聚关键数据、展示治理态势、追踪风险和闭环，不要把所有端口能力都搬进来。

- 结论3：犬只是监管主对象，主手机号是积分和权益账户。  
  不单独做犬主 tab。  
  犬只档案中展示主手机号、绑定状态、积分摘要、附属联系方式等信息。  
  狗牌 / 设备与犬只由政府侧统一绑定，犬主人端只做主手机号与狗狗 / 狗牌的绑定或认领。监管端负责查看绑定状态和异常兜底，而不是作为犬主手机号绑定的主流程入口。

- 结论4：积分必须单独成模块。  
  因为这个生态不是纯处罚监管系统，而是文明养犬服务生态。生态分析里明确提到，不应只强调“发现问题 → 监管处理 → 处罚约束”，而应转向“正向行为 → 获得收益 → 数据沉淀 → 更好服务 / 更好治理”。  
  所以积分管理要支撑规则、账户、流水和激励效果，而不只是一个“修改积分”的后台表格。

- 结论5：一线端和监管端职责要分开。  
  一线巡查端一期已经明确只聚焦“查犬 + 看位置 + 处理任务”，不做复杂地图巡查、不做权限管理、不展示设备原始数据、不接入疫苗体检记录。  
  因此监管端可以承担更完整的全局态势、地图监管、数据汇总和积分管理，但不要反过来把一线端变复杂。

## 5. 还没定的点

- 待定1：地图工作台一期具体做到什么深度。  
  已确定地图是主要工作台之一，但需要控制边界。  
  待定内容包括：

```text
是否展示城管实时位置，还是只展示任务区域 / 负责辖区
是否展示服务商 / 合作商点位
重点区域 / 禁遛区域一期是否做
是否做风险热力，还是只做事件点位
是否在地图中承接事件处理，还是只做跳转详情
```

建议一期只做图层展示 + 点位详情 + 简单筛选，不做复杂轨迹和自动调度。

- 待定2：对象档案里的服务商 / 合作商展示深度。  
  监管端可以看服务商位置、合作状态、积分核销次数、服务覆盖情况。  
  但深度经营分析应该留给运营后台，因为运营后台本来就承担积分经营分析和终端服务商家管理。  
  待确认一期是否只展示：

```text
服务商名称
类型
位置
合作状态
积分核销次数
最近核销时间
```

- 待定3：积分管理一期是否允许手动调整积分。  
  来源里监管后台职责有“积分管理，包括查看、修改等”。  
  但手动调整积分涉及权益账户，风险较高。  
  待定是否一期只做规则、账户、流水查看，还是开放“手动调整”。  
  若开放，必须有：

```text
权限控制
调整原因
操作日志
调整记录
```

- 待定4：AI 分析 / 风险功能一期是真能力还是演示能力。  
  治理驾驶舱里会放 AI 分析 / 风险提醒，但一期要明确是：

```text
基于已有规则的风险提示
还是基于算法识别的智能分析
```

如果算法数据未成熟，一期不要把 AI 说得太满。别让 AI 背一个产品没落地的锅，AI 也有尊严，虽然不多。

## 6. 不要走的方向

- 不要做成：  
  不要做成“全功能后台”或“菜单大杂烩”。  
  不要因为监管后台原始职责里有很多内容，就把管理员、权限、设备、犬主、犬只、事件、城管、服务商、数据看板全部拆成一级菜单。  
  一期坚持四页：

```text
治理驾驶舱
地图工作台
对象档案
积分管理
```

- 不要强调：  
  不要只强调异常、报警、处罚、违规。  
  监管端要体现“治理可视、风险可控、任务可追、成效可汇报”，同时也要体现正向治理，比如积分参与率、文明行为、服务核销、设备在线改善、异常下降趋势。生态分析里也明确建议监管后台不要只看异常数、报警数、违规数，而应增加文明遛狗、积分参与、异常处理完成率、服务商积分核销等正向指标。

- 不要纳入一期：

```text
复杂权限系统
复杂流程引擎
完整事件中心独立页面
独立设备管理一级页面
独立犬主管理一级页面
独立服务商经营后台
复杂轨迹回放
复杂热力分析
自动任务调度
算法模型配置平台
宠物医院 / 疫苗体检记录
完整绩效考核排名
复杂经营分析
```

特别注意：疫苗体检记录不要做。之前一线端方案已经明确当前暂无可靠数据来源，拿不到的数据不要硬做页面。
